Overview

⚡ How LISA Works: Simple Version

No technical background required. This is how LISA handles every message from end to end.

📚 The simplest analogy: LISA is like a new employee who has read every single document Lark Finserv has ever written: all FAQs, all lender agreements, all product manuals: and can recall the right page in under 2 seconds. The key difference: this employee works 24/7, never misquotes a number, and can serve 1,000 partners simultaneously.
1
Partner asks a question
"What is the minimum portfolio value for LAMF?": typed into the chat window on the web portal.
2
LISA classifies the intent
AI instantly categorizes it as a product question (not content creation or analytics). This determines which prompt template and tools are used next.
3
LISA searches the knowledge base
Scans all of Lark's documents and picks the 5 most relevant paragraphs in milliseconds. This is called RAG: Retrieval Augmented Generation.
4
GPT-4o reads only those paragraphs and answers
The AI sees only the 5 retrieved paragraphs and forms an answer. It cannot make up numbers or policies: it can only use what was retrieved.
5
Response appears on screen
Partner sees the answer in ~1.5 seconds. The full conversation is saved so LISA remembers context in follow-up messages.

Eligibility Check
Client's portfolio is checked against the eligible fund list. Minimum value Rs. 50,000. System checks lien marking eligibility with CAMS / KFintech in real time.

Technical

🔘 RAG Pipeline: How LISA Retrieves Knowledge

RAG (Retrieval Augmented Generation) is the core technology that prevents hallucination and makes LISA's answers trustworthy.

📚 Why RAG? Without RAG, GPT answers from its 2023 training data: it doesn't know your interest rates, your lenders, your products. With RAG, LISA first retrieves the exact paragraphs from Lark's documents, then GPT answers only from those paragraphs. The LLM becomes a reader, not a guesser.

Phase 1: Ingestion (run once before launch)

backend/chatbot/rag/ingest.py
STEP 1: LOAD DOCUMENTS
  .txt files  → TextLoader  (reads raw text line by line)
  .csv files  → CSVLoader   (reads row by row, each row = document)
  Each doc gets metadata: { source, topic, partner_type, file_type }

STEP 2: CHUNK (RecursiveCharacterTextSplitter)
  Settings:
    chunk_size    = 500 characters
    chunk_overlap = 100 characters

  Why 500 chars? Small enough to stay within GPT's context window.
  Why overlap? Sentences that span chunk boundaries are preserved.

  One document (50,000 chars)  →  ~100 chunks of 500 chars each

  Split priority order (tries larger breaks first):
    1. \n====   major section headers
    2. \n----   minor section headers
    3. \n\n     paragraph breaks
    4. \n       new lines
    5. .        sentence ends
    6. (space)  word breaks  ← last resort only

STEP 3: EMBED (OpenAI text-embedding-3-large)
  Each chunk → API call → [0.023, -0.412, 0.891, ...] (3072 numbers)
  Chunks with similar meaning → similar numbers (close in vector space)
  This is semantic search: meaning-matching, not keyword-matching.

  "What is LAMF rate?"         → similar vector as → "Interest rate 10.5%"   ✓
  "loan against mutual funds"  → similar vector as → "LAMF product details"  ✓

STEP 4: SAVE FAISS INDEX
  FAISS.from_documents(all_chunks, embedding_model)
  Saves two files:
    vectorstore/index.faiss   ← all 3072-dim vectors, compressed + indexed
    vectorstore/index.pkl     ← original text + metadata for each chunk

Phase 2: Query Time (every user message)

backend/chatbot/rag/retriever.py + reranker.py
STEP 5: EMBED QUERY
  User's message → same OpenAI embedding model → query vector

STEP 6: FAISS SIMILARITY SEARCH
  faiss_index.similarity_search_with_score(query_vector, k=15)
  Returns top 15 chunks by cosine distance in milliseconds
  Score closer to 0 = more similar

STEP 7: CROSS-ENCODER RERANK
  Model: ms-marco-MiniLM-L-6-v2
  For each of the 15 chunks, scores the (query, chunk) pair:
    (query, chunk_1) → 0.92  ← most relevant
    (query, chunk_2) → 0.71
    (query, chunk_3) → 0.45
    (query, chunk_4) → 0.38
    (query, chunk_5) → 0.21
    ...
  Re-sorted → top 5 most relevant chunks selected

STEP 8: INJECT INTO LLM PROMPT
  Top 5 chunks → inserted as [CONTEXT DOCUMENTS] in prompt
  GPT-4o-mini answers ONLY from context
  If context has no answer → LISA says "I don't have that info"
💡 Why two stages? FAISS is extremely fast but gives approximate results. The cross-encoder is precise but slow. Together: FAISS narrows 500+ chunks to 15 in under 1ms, then the cross-encoder precisely scores only those 15. You get both speed and accuracy.
Technical

🖸 LangGraph: The AI Workflow

Every user message passes through a 5-node graph. Each node adds data to a shared state object and passes it forward.

📦 Analogy: Think of GraphState as a package that travels through 5 departments. Each department opens it, adds something (intent label, retrieved docs, AI response, saved message ID), reseals it, and sends it to the next. At the end, the API unpacks it and sends the response to the partner.
1
Intent Classifier

LLM reads the user's message and returns JSON: {"intent": "query_support", "confidence": 0.97}. Routes to the correct prompt and tools based on intent.

↓
2
RAG Retrieval

HybridRetriever embeds the query → FAISS returns top 15 chunks → CrossEncoderReranker scores them → top 5 added to state["reranked_documents"].

↓
3
LLM Response

Selects prompt template based on intent. Injects top 5 docs + last 6 conversation messages. GPT-4o-mini generates response. If a tool is needed, sets state["tool_name"].

↓
3b
Tool Execution (conditional)

If tool_name is set: calls FAQ Retriever, Content Generator, or Partner Analytics. Tool output appended to response. Skipped if no tool needed.

↓
4
Memory Update

Saves both user and assistant messages to SQLite. Auto-sets session title from first message. Returns message_id (UUID): used by the frontend for feedback thumbs up/down.

↓
5
Trace Logger

Logs to LangSmith: intent classification, number of docs retrieved, LLM latency, total end-to-end latency. Full observability for every single message.

GraphState: What Flows Through

GraphState = {
  # Set by API route before graph starts
  "query":                "What is the minimum portfolio value for LAMF?",
  "user_id":              "uuid-logged-in-user",
  "session_id":           "uuid-chat-session",
  "conversation_history": [{"role": "user", "content": "..."}, ...],  # last 20 msgs

  # Added by Node 1: Intent Classifier
  "intent":               "query_support",     # or content_generation / behaviour_discovery
  "intent_confidence":    0.97,
  "is_in_scope":          True,

  # Added by Node 2: RAG
  "retrieved_documents":  [ ... ],             # top 15 raw FAISS chunks
  "reranked_documents":   [ ... ],             # top 5 after cross-encoder

  # Added by Node 3: LLM
  "response":             "Minimum portfolio value for LAMF is Rs. 50,000...",
  "tool_name":            None,               # or "analyze_partner_behavior" if needed

  # Added by Node 3b: Tool (only if tool_name set)
  "tool_output":          "...",

  # Added by Node 4: Memory
  "memory_updated":       True,
  "message_id":           "uuid-saved-message",  # sent to frontend for feedback

  # Added by Node 5: Trace
  "trace_metadata":       { "latency_ms": 1240, "intent": "query_support", "num_docs": 5 },

  # Error (if anything fails)
  "error":                None
}

🔒 Security

Every known attack surface and how it is mitigated in LISA.

ThreatHow It's Handled
API Key ExposureKeys in .env only: .gitignore prevents commit. Never hardcoded in any Python file.
Password StorageScrypt hash (cost factor 32768). Never stored as plain text. Not reversible.
Unauthorized API Access@login_required on every /api/* route. Returns 401 JSON (not a redirect) for API consumers.
XSS AttacksAll markdown rendered via marked.js is sanitized with DOMPurify before DOM insertion. Prevents script injection.
Database Exposureinstance/ folder in .gitignore. Database never pushed to GitHub.
Session HijackingFlask SECRET_KEY (64+ random chars). HttpOnly secure cookies. Session expires on logout.
Prompt InjectionUser input always passed as named variable, never string-concatenated into prompt template.

Data Leakagedata/ folder git-ignored. Business documents never go to GitHub or any public repository.

✏ How to Write Documents for RAG

Documents are split into 500-character chunks. Each chunk must be self-contained and include the key numbers explicitly.

✓ Write this way❌ Avoid this
"LAMF interest rate starts at 10.5% per annum for equity funds.""The rate is mentioned in the table below."

============================
BFL (Bajaj Finance): LAMF Product Details
============================
BFL (Bajaj Finance Limited) offers Loan Against Mutual Funds (LAMF)
through the Lark Finserv platform.

Interest rate: 10.5% to 11.5% per annum (floating rate, subject to change).
Minimum loan amount: Rs. 25,000.
Maximum loan amount: Based on LTV of pledged portfolio.
LTV ratio: Up to 50% for equity mutual funds, up to 80% for debt funds.

============================
BFL LAMF Process
============================
The BFL LAMF process is fully digital. Steps:
1. Portfolio eligibility check via CAMS/KFintech
2. Digital KYC via CKYC lookup or Aadhaar OTP
3. Lien marking on mutual fund units via CAMS
4. E-sign via Aadhaar OTP
5. NACH e-mandate registration
6. Disbursement to registered bank account

TAT: Same day disbursement (T+0) for complete digital applications.
Processing fee: 0.5% of loan amount.

============================
BFL LAMF Eligibility
============================
Eligible fund categories: Equity funds, debt funds, hybrid funds, ETFs, index funds.
Ineligible: ELSS funds (lock-in), closed-ended funds.
Minimum portfolio value: Rs. 50,000.
KYC: Must be Aadhaar/PAN linked.

🛠 Fresh Setup: Step by Step

Complete guide to running LISA from scratch on a new machine.

Step 1: Clone and Install

git clone https://github.com/larkfinserv/LISA-RAG.git
cd LISA-RAG
pip install -r requirements.txt

Step 2: Create .env File

# Create this file manually: never commit it to git
OPENAI_API_KEY=sk-your-openai-key-here
SECRET_KEY=generate-a-long-random-string-here

# Database (SQLite for dev, change to PostgreSQL for production)
DATABASE_URL=sqlite:///instance/lisa.db

# Optional: LangSmith tracing (get free key at smith.langchain.com)
LANGCHAIN_TRACING_V2=true
LANGCHAIN_API_KEY=ls-your-langsmith-key
LANGCHAIN_PROJECT=LISA-RAG

Step 3: Add Knowledge Documents

# Drop .txt and .csv files into the data/ folder
# Minimum required to start:
data/
  AI_B2C_RAG_Optimized.txt

  ABFL Tata Offline LAS.txt
  Lark_Finserv_RAG_Notes.txt
  Complete_Training_Dataset_Labeled.csv
# See "What Docs LISA Needs" section for full list

Step 4: Create Database Tables

python backend/scripts/create_db.py
# → "Database tables created successfully!"
# Creates instance/lisa.db with 4 tables

Step 5: Create Login Accounts

python backend/scripts/seed_users.py
# Creates: admin/admin123 · demo_mfd/demo123 · demo_wealth/demo123

Step 6: Build the FAISS Index

python backend/scripts/ingest_documents.py
# Loads all files from data/
# Chunks, embeds via OpenAI, builds FAISS index
# Saves: vectorstore/index.faiss + vectorstore/index.pkl
# Time: ~2-5 minutes · Cost: ~$0.02 one-time · Re-run only when docs change

Step 7: Start LISA

KMP_DUPLICATE_LIB_OK=TRUE python app.py
# → http://localhost:5001
# Login: admin / admin123

📄 Adding New Knowledge to LISA

1
Write the document
Create a .txt file. Use ===== section headers. State all numbers explicitly in each section. Keep paragraphs short. See the format guide above.
2
Add to data/ folder
Place the file in data/. Name it descriptively: BFL_Product_Sheet.txt, Partner_Payout_Guide.txt, etc.
3
Re-run ingestion
python backend/scripts/ingest_documents.py: rebuilds the entire FAISS index with all documents including the new one. Takes 2-5 minutes.
4
Restart the server
FAISS is loaded into memory at startup. Restart: Ctrl+C then python app.py. LISA now knows the new document immediately.
